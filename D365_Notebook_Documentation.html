1. Overview

This notebook ingests data from Microsoft Dynamics 365 (Dataverse) into Databricks Delta tables in Unity Catalog. It reads CSV files exported by Azure Synapse Link to ADLS Gen2, joins them with the model.json schema file (Synapse Link's schema metadata), applies inserts/updates/deletes, and writes Delta tables — exactly what the managed connector does.

Why use this notebook instead of the managed connector?

  • Beyond 250 tables: The managed connector caps at 250 tables per pipeline. This notebook handles 600+ via batching.
  • Full control: You own the code, can debug, customize, and integrate with your workflows.
  • Resume from failure: If a run fails at table #237 of 600, next run picks up from #237 automatically.
  • Audit trail: Every run logged to a Delta table for compliance and troubleshooting.

When to use the managed connector instead?

  • Small deployments (< 100 tables) where the 250 limit is not a concern.
  • You prefer a Databricks-managed service with a UI.
  • You don't need custom retry/resume logic beyond what Databricks provides.

6. Failure Recovery (Rerun Process)

This is the most important section. The notebook is designed so that rerunning after a failure is trivial and always safe.

How Resume Works

Cursor state is stored durably in Delta table: catalog.d365_control.d365_ingestion_cursor Every table has one row tracking: ┌──────────┬────────┬─────────────────┬──────────────────┐ │ table │ status │ last_vernumber │ last_folder_ts │ ├──────────┼────────┼─────────────────┼──────────────────┤ │ account │ SUCCESS│ 1234567890 │ 2026-07-09T10:00 │ │ contact │ SUCCESS│ 9876543210 │ 2026-07-09T10:00 │ │ opportu…│ FAILED │ 5555555555 │ 2026-07-09T09:00 │ │ lead │ RUNNING│ 3333333333 │ 2026-07-09T09:00 │ └──────────┴────────┴─────────────────┴──────────────────┘ On next run: 1. Recover stuck RUNNING tables → mark as FAILED 2. Skip SUCCESS tables (within resume_skip_minutes window) 3. Retry FAILED tables FIRST (from saved cursor) 4. Process never-run tables 5. Refresh stale SUCCESS tables

Recovery Scenarios

ScenarioWhat HappenedWhat You Do
A Cell failed, cluster still running Re-run Section 16 only — results = run_ingestion()
B Cluster crashed / restarted Re-run all cells top-to-bottom (reload functions)
C Want to retry ONLY failed tables retry_failed_only() — no config edit needed
D Tables stuck in RUNNING reset_stuck_running() — instant recovery
E Scheduled Databricks Job Do nothing — next scheduled run auto-resumes
F Schema changed in D365 Set force_full_refresh = True, run once, reset to False

Example: 600-Table Failure Recovery

Scenario: You start ingestion of 600 tables. Run 1 processes tables 1-236 successfully, then fails at table 237 due to a transient ADLS issue. What do you do?

Option 1 — Just re-run the cell:

# Simply re-run Section 16 (results = run_ingestion())
# The notebook will:
# 1. Recover any stuck RUNNING tables
# 2. Skip tables 1-236 (they're SUCCESS within resume_skip_minutes)
# 3. Retry table 237 FIRST (from cursor)
# 4. Continue with tables 238-600

Option 2 — Check what will happen first:

preview_ingestion()
# See exactly what will be processed vs skipped
results = run_ingestion()

Option 3 — If you fixed a specific issue and only want to test failed tables:

retry_failed_only()
# Retries ONLY the FAILED tables, skips all others

Data Flow — Per Table

For each table (e.g. "account"): ADLS: /account/Snapshot/2026-07-09T00:00:00/account.csv ← full state /account/Changelog/2026-07-09T00:05:00/changelog.csv ← changes /account/Changelog/2026-07-09T00:10:00/changelog.csv ← more changes Notebook: First run: read Snapshot → write full Delta table Later runs: read new Changelog folders since cursor → MERGE into Delta Cursor updates after each folder processed → resume-safe.

SQL Query Examples (Section 18)

Get current state of a table:

SELECT * FROM main.d365_raw.account LIMIT 100;

SCD2 — current records only:

SELECT * FROM main.d365_raw.account WHERE __CURRENT = true;

SCD2 — full history for a specific record:

SELECT * FROM main.d365_raw.account
WHERE accountid = 'your-guid-here'
ORDER BY __START_AT;

Resolve Picklist integer to label via OptionSetMetadata:

SELECT a.accountid, a.name,
       a.industrycode AS industry_code,
       osm.LocalizedLabel AS industry_label
FROM main.d365_raw.account a
LEFT JOIN main.d365_raw.optionsetmetadata osm
  ON osm.OptionSetName = 'account_industrycode'
 AND osm.Option = a.industrycode
 AND osm.LocalizedLabelLanguageCode = 1033;

Multi-select Picklist — explode into rows:

SELECT accountid, CAST(code AS INT) AS option_code
FROM main.d365_raw.account
LATERAL VIEW EXPLODE(SPLIT(industrycodes, ',')) AS code;

Lookup join — get related record:

SELECT a.accountid, a.name, c.fullname AS primary_contact
FROM main.d365_raw.account a
LEFT JOIN main.d365_raw.contact c
  ON a.primarycontactid = c.contactid;

9. Scheduling in Production

Schedule the notebook as a Databricks Workflow:

1 Workflows → Create Job

2 Task type: Notebook, select D365_Connector_Notebook_v5

3 Set schedule (e.g. every 1 hour to match Synapse Link export interval)

4 Use a job cluster for cost efficiency

5 Enable email notifications on failure

Why scheduling works automatically:
  • Cursor state persists across runs in Delta
  • Each scheduled run picks up from prior cursor
  • If run N fails, run N+1 automatically resumes
  • Run history table gives full audit trail

10. Troubleshooting

ErrorCauseFix
Cannot read model.json Synapse Link hasn't exported yet, or ADLS auth wrong Verify Synapse Link status is Active; check Storage Blob Data Contributor role
versionnumber missing from changelog Synapse Link not configured to export versionnumber Reconfigure Synapse Link; set force_full_refresh = True and re-run once
Type mismatch / Cannot cast D365 changed column data type Set force_full_refresh = True for affected table only
Table stuck in RUNNING Cluster crashed mid-processing Call reset_stuck_running() or wait stuck_running_hours
No data found Synapse Link paused, or wrong tables selected Resume Synapse Link in Power Apps; verify tables in export config
Delta MERGE conflict Two runs writing to same table simultaneously Ensure only one run active; check for stuck RUNNING status
ADLS 403 Forbidden Service principal missing storage role Grant Storage Blob Data Contributor role in Azure IAM

11. Known Limitations

These limitations come from the underlying Synapse Link + Dataverse architecture and apply equally to the managed connector:

Latency Data freshness bounded by Synapse Link export interval (5-15 min minimum)
Attachments File metadata only, no binary content (use D365 Web API separately)
Schema Changes Require full refresh (force_full_refresh = True)
Backfill No auto-backfill for new columns added after ingestion started
Downtime If Synapse Link stops, missed changes require full refresh
F&O Virtual Entities May take up to 15 min for schema changes to synchronize
Source Table Delete Deleting a table in D365 does NOT delete the destination Delta table (manual cleanup needed)
Duplicate Names Cannot ingest two tables with same name in one pipeline — use separate schemas
