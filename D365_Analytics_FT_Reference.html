§ 01
Overview

Three questions this document answers.

If you came here from a search result, these are the questions you probably had. Jump straight to the section that matches.

01
What are the EntityUtil and AdditionalScripts, really?
Migration tooling built by Microsoft's FastTrack team to help customers move off BYOD. They recreate BYOD-style entity views on top of the raw tables that Synapse Link now exports. Not a runtime — a migration aid.
02
Where do they run, and where do they not run?
PowerShell on a local Windows machine, connecting outbound to a D365 sandbox and a target SQL database (Synapse, Fabric, or Azure SQL). They do not produce Delta tables. They do not run on Spark. They are not a Databricks tool.
03
How does this map to Raw / Silver / Gold / OBT?
Their logic is a blueprint. Inheritance joins map to Silver. Entity joins map to Gold. The discovery SQL tells you which child tables to add to Synapse Link. But you rewrite the transformations in Spark; you do not run the .ps1 files.
§ 02
Origin

Why these scripts exist at all.

BYOD delivered pre-joined entity views. Synapse Link delivers raw tables. The scripts bridge that gap for customers still consuming entity-shaped data.

BYOD — legacy
Pre-joined entity views, scheduled batch
  • Scheduled export from F&O to a customer-managed Azure SQL DB
  • Delivered flat data entities — CustCustomerV3Entity joined ten tables at source
  • Reporting consumed the entity directly with no reconstruction
  • Being deprecated: locks the source, poor incremental support
Synapse Link — current
Raw tables, near-real-time change tracking
  • Change-tracked replication to Azure Data Lake Storage
  • Individual base tables — not entities
  • Consumer responsible for reconstructing entity shapes
  • Inherited tables (DirPartyTable) land as fragments

The scripts sit between the raw Synapse Link output and the reports that expect BYOD-style shapes. For a lakehouse team, the same reconstruction happens in Databricks notebooks — the scripts are useful as a blueprint, not as a runtime.

Where the scripts run (and don't).

Everything runs on a local Windows machine. From there, the scripts reach out to two different databases — never to Databricks.

PowerShell + Azure CLI + SSMS
All scripts execute here; nothing runs server-side.

dependencies.json Local file — the pivot point
Target — SQL database
Synapse / Fabric / Azure SQL
Contacted by Steps 2, 3, 5 via Azure AD auth. This is where entity and inherited-table views land.

The sequence of steps

The menu numbers steps 1 through 5, but on Fabric and Serverless-replica targets you must run Step 5 before Step 2. Step 2's entity views depend on the inherited-table views Step 5 creates.

0
Discovery — get_derivetables.sql
SSMS · reads sandbox
Enumerate the D365 inheritance hierarchy. Take the output back to Power Platform admin to add every child table to your Synapse Link profile before continuing.

PowerShell · reads sandbox · writes dependencies.json
Recursively walk entity dependencies for the entities you listed in config. Everything downstream reads from the JSON file this step writes.

PowerShell · reads target · writes VIEWs

PowerShell · reads dependencies.json · writes VIEWs

Silver — the seven transformations

This is the work. Every downstream layer depends on Silver being correct.

Filter soft deletes
WHERE IsDelete = 0 — removes records logically deleted upstream.
Deduplicate on recid
Keep the row with the highest versionnumber per recid. Synapse Link may land multiple versions during incremental sync.
Reconstruct inherited tables
LEFT OUTER JOIN each parent to all its children on recid. The join statements from Step 0 are your blueprint.
Resolve enum labels
Add a _label column beside each enum integer by joining to srsanalysisenums.
Clean column names
Strip trailing underscores from reserved words (LEVEL_ → level). Drop Synapse Link infrastructure columns.
Cast data types
String → Timestamp / Decimal / Integer. Standardise dataareaid casing to upper case.
Explode financial dimensions
Pivot the DimensionAttributeValueCombination hierarchy into named columns per your chart of accounts.
§ 06
Critical

Silver is not an entity.

This is the single most common trap for teams transitioning from BYOD. Reconstructing an inherited table is not the same job as recreating a D365 entity. They live in different layers.

Common confusion

Different joins, different layers.

An inherited-table join makes one D365 table whole again. An entity join stitches many different tables into a business shape.

Inherited-table join · Silver
silver_dirpartytable
  • Joins a table to its own inheritance children
  • All rows share the same recid
  • Purpose: reassemble one D365 table
  • = dirpartytable + companyinfo + dirperson + dirorganization
Analogue to EntityUtil Step 5
Entity-style join · Gold
dim_customer
  • Joins entirely different tables
  • Different recids, different domains
  • Purpose: build a business dimension or fact
  • = custtable + silver_dirpartytable + custgroup + address
Analogue to EntityUtil Step 2

Why keep them separate

If Silver contained entities, silver_CustCustomerV3Entity, silver_VendVendorV2Entity and silver_HcmWorkerEntity would each contain their own copy of the DirPartyTable join. Party address logic would live in three places. Any change would require three edits.

Instead, keep Silver granular: one silver_dirpartytable, referenced by every Gold dimension that needs party attributes. Change once, propagate everywhere.

§ 09
Guidance

Anti-patterns we avoid.

Five patterns that look tempting from a BYOD background but cause real problems in a lakehouse.

✕
Recreating BYOD entities as Silver tables
Duplicates join logic. Party attributes end up in three Silver tables at once. Change once, break twice.
✕
Scheduling the .ps1 scripts as a Databricks job
The scripts target Synapse or Fabric SQL. They don't write Delta. They don't run on Spark. Wrong tool for the job.
✕
Skipping get_derivetables.sql at the start
Child tables silently missing from Synapse Link. Every downstream layer will have gaps you cannot see until a report user asks why a customer name is blank.
✕
Filtering IsDelete in Raw
Destroys the audit trail. When a business-rule error is discovered later, reprocessing from history becomes impossible.
✕
Connecting Power BI directly to Silver
Report authors will reproduce entity joins in DAX. Push those joins into Gold or OBT once — not into every report forever.
§ 10
Close

Four things to remember.

01
The scripts are Synapse Link tooling
They exist to bridge BYOD to Synapse Link. None of them run in a lakehouse. Do not schedule them as Databricks jobs.
02
Their real value is as a blueprint
get_derivetables.sql tells you which tables to add to Synapse Link. dependencies.json tells you which entity joins to reproduce in Spark.
03
Silver is not an entity
Silver makes each D365 table whole again — that's Step 5 work. Gold joins many Silver tables into business shapes — that's Step 2 work. Confusing the two creates duplicated join logic and unmaintainable pipelines.
04
OBT is where Power BI plugs in
Not Silver, not Gold. Pre-join at load time in a wide table; keep reports thin.
